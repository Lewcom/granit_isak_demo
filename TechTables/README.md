# TechTables

Технічні таблиці для команди.
Інтерфейсний файл для клієнта лишається: `granit-isak-crm-demo.html`.

## Файли

- `components_catalog.json`
  Повний каталог компонентів (Single + Double) + мапінг скорочень (`aliases`) + структура кольорів/параметрів/цін.

- `single_base.json`
  Базові правила і сімейства компонентів для Single.

- `single_validation_2026-03-01.md`
  Протокол звірки Single 120/220 по клієнтських фото (OK/розбіжності/питання).

- `double_base.json`
  Каталог Double 220x220 (синхронізовано з UI-фільтром `mtype=double`).

## Джерело істини

Поточний режим: **manual mirror** (дублювання).

- CRM (`PT` + typed-об'єднання в `granit-isak-crm-demo.html`) = робочий інтерфейсний файл.
- `TechTables/*.json` = технічний еталон для команди, звірка, нормалізація назв.

## Модель даних у UI

- В UI використовується один об'єднаний каталог (single+double).
- Поле рядка `mtype` (`single`/`double`) фільтрує список компонентів, кольорів, значень і відповідних цін.
- `Рамка` має розширені ширини і всі кольори для обох типів.

## Нормалізація скорочень

Використовувати канонічні назви:

- `чор/сір`, `чорна/сіра` -> `чорний`, `сірий`
- `дідкович` -> `дідковичі`
- `ток/межир` -> `токівський`, `межирічка`
- `зелен`, `зелена` -> `зелений`
- `ін.колір` -> `інший колір`

## Checklist синхронізації

Після зміни `PT` у CRM:

1. Оновити `components_catalog.json` (і `single_base.json`, якщо змінили правила/сімейства Single).
2. Перевірити `aliases` (чи не з'явились нові скорочення).
3. Якщо зміна стосується подвійного прайсу — оновити `double_base.json`.
4. Перевірити, що назви компонентів/кольорів узгоджені в усіх файлах.

Після оновлення TechTables:

1. Перевірити, що відповідні позиції присутні в `PT`.
2. Якщо треба, внести дзеркальну правку в `granit-isak-crm-demo.html`.

## Поточний статус

- Single/Double працюють через фільтр `mtype` у одному UI-каталозі.
