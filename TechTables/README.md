# TechTables

Технічні таблиці для команди.
Інтерфейсний файл для клієнта лишається: `crm_monuments (1).html`.

## Файли

- `components_catalog.json`
  Повний поточний каталог компонентів (Single), мапінг скорочень (`aliases`), структура кольорів/параметрів/цін.

- `single_base.json`
  Базові правила і сімейства компонентів для Single.

- `double_base.json`
  Окрема чернетка каталогу для Double 220x220 на основі клієнтських фото (`2-1`, `2-2`).

## Джерело істини

Поточний режим: **manual mirror** (дублювання).

- CRM (`PT` в `crm_monuments (1).html`) = робочий інтерфейсний файл.
- `TechTables/*.json` = технічний еталон для команди, звірка, нормалізація назв.

## Нормалізація скорочень

Використовувати канонічні назви:

- `чор/сір`, `чорна/сіра` -> `чорний`, `сірий`
- `дідкович` -> `дідковичі`
- `ток/межир` -> `токівський`, `межирічка`
- `зелен`, `зелена` -> `зелений`
- `ін.колір` -> `інший колір`

## Checklist синхронізації

Після зміни `PT` у CRM:

1. Оновити `components_catalog.json` (і `single_base.json`, якщо змінили правила/сімейства).
2. Перевірити `aliases` (чи не з'явились нові скорочення).
3. Якщо зміна стосується подвійного прайсу — оновити `double_base.json`.
4. Перевірити, що назви компонентів/кольорів узгоджені в усіх файлах.

Після оновлення TechTables:

1. Перевірити, що відповідні позиції присутні в `PT`.
2. Якщо треба, внести дзеркальну правку в `crm_monuments (1).html`.

## Відкриті питання по Double

- `Рамка на подвійний` у джерелі без розбивки по кольорах (потрібне підтвердження логіки).
- Потрібно фінально погодити, чи `double_base.json` лишається окремим каталогом або стане частиною єдиного `PT`.
